# AI 竞品情报系统 — 项目工程介绍

> **版本**：3.0.0 | **技术栈**：Python 3.13 · Streamlit · LangGraph · OpenRouter · Tavily/DuckDuckGo · APScheduler · SQLite
>
> **目标**：输入竞品名称，自动完成从数据采集到战略分析的完整竞品报告生成，支持增量对比、批量矩阵、定时监控与邮件推送

---

## 一、项目背景与定位

### 解决什么问题

传统竞品分析需要人工花费数小时在网上搜集信息、整理报告。本项目将这个过程自动化：

1. 自动在互联网上搜索竞品的官网信息、新闻动态、用户评价、招聘信号
2. 使用 LLM 将原始数据提炼成结构化分析
3. 输出包含功能矩阵、SWOT 分析、战略建议的完整 Markdown/HTML 报告
4. 全程约 2~3 分钟，成本约 $0.001~0.003
5. **增量分析**：再次分析同一竞品时，自动加载历史数据，LLM 重点关注变化
6. **批量对比**：一次输入 5 个竞品，并发分析后自动生成横向对比矩阵
7. **定时监控**：开启后每天/每周自动重跑分析，变更通过侧边栏徽章 + 邮件双通道推送

## 三、技术选型说明

| 层级 | 技术 | 选择原因 |
|------|------|---------|
| **UI** | Streamlit | 纯 Python 构建 Web 界面，无需前端知识，一行命令启动 |
| **LLM** | OpenRouter API | 一个接口聚合所有主流模型，兼容 OpenAI SDK，随时切换 |
| **搜索主路** | Tavily API | 专为 AI Agent 设计，返回干净结构化结果，有免费额度（1000次/月） |
| **搜索兜底** | DuckDuckGo | 无需 API Key，Tavily 失败时自动切换，保障可用性 |
| **爬取** | Crawl4AI | 开源，自动将 HTML 转 Markdown，省 Token，可选安装 |
| **Agent 编排** | LangGraph | TypedDict 类型安全状态、条件边、MemorySaver 断点续跑 |
| **知识持久化** | SQLite | 零运维，单文件，四张表共享同一 history.db |
| **定时任务** | APScheduler | BackgroundScheduler 在 Streamlit 进程内后台运行，重启自动恢复 |
| **邮件通知** | Python smtplib | 标准库，零额外依赖，支持任意 SMTP 服务商 |
| **配置管理** | python-dotenv | `.env` 文件管理密钥，防止误提交 |

### 4.2 单竞品完整执行流程

```
用户输入「Notion」并点击「开始分析」
       │
       ├─ KnowledgeStore.get_prior_context("Notion")
       │   → 若有历史数据，UI 显示「📚 知识库：已有历史数据（2026-xx-xx）」
       │
       ▼
【Phase 1 - researcher（asyncio.gather 并发，含增量注入）】
  ├── 任务A: 搜索官网信息 + Crawl4AI 爬取首页
  ├── 任务B: 搜索近30天新闻
  ├── 任务C: 搜索用户评价（G2/ProductHunt）
  └── 任务D: 搜索招聘信号（LinkedIn）
  ↓（ResultFilter → DiversityFilter 双重过滤）
  每路搜索均 require_keyword=competitor，过滤无关结果
  构建编号来源列表（[1]…[N]），注入 raw_data 顶部
  LLM 汇总：原始数据 + prior_context → 带 [N] 引用的调研摘要
  _strip_code_fence() 后处理：去除意外的 ```markdown 包裹
       │
       ▼
【Phase 2 - analyst（顺序）】
  输入：researcher 摘要 + 新闻/招聘原始数据
  LLM 输出：融资历史、营收规模、近期重大事件（约600字）
       │
       ▼
【Phase 3 - product（顺序）】
  LLM 输出：功能对比矩阵（Markdown 表格，约500字）
       │
       ▼
【Phase 4 - strategist（顺序）】
  LLM 输出：SWOT 分析 + 3~5 条可执行战略建议（约800字）
       │
       ▼
【Phase 5 - evaluator（反思循环）】
  6 维度评分（每维0~2分，总分映射到1~10）
  score ≥ 7 → 直接进入报告生成
  score < 7 → 返回 gaps + supplement_queries
              ↓
         并发补充搜索（针对缺口的英文查询词）
              ↓
         追加补充数据到 research_data
              ↓
         重跑 strategist（仅此 1 次）
       │
       ▼
【IRDocument 中间表示 → ReportGenerator 渲染】
  from_agent_outputs() → validate()（检测薄章节） → to_markdown()
  报告头部嵌入质量徽章：「质量评分：7/10 ✓」
  报告底部追加「参考来源」章节（可点击 Markdown 链接）
       │
       ▼
【KnowledgeStore.extract_and_save()】
  保存 summary / analysis / feature_matrix 三类事实
  下次分析同名竞品时自动加载为 prior_context
       │
       ▼
【storage.py → history.db + UI 展示 + 下载按钮（MD/HTML）】
```

### 4.3 搜索结果三级过滤漏斗

```
Tavily 原始返回（8~11 条）
       │
       ▼ ResultFilter.filter(require_keyword=competitor)
  ① score 降序排列
  ② URL 去重
  ③ 内容长度 < 80 字符 → 丢弃
  ④ 域名黑名单（pinterest/amazon 等）→ 丢弃
  ⑤ 同域名 > 2 条 → 截断
  ⑥ title+content 不含竞品名 → 丢弃（过滤无关新闻）
       │
       ▼ DiversityFilter.select(target_k=5, sim_threshold=0.45)
  n-gram Jaccard 贪心去重，剔除内容高度相似结果
       │
       ▼ 最终 5 条高质量、高相关、多样化结果

典型漏斗：8 条 → 质量过滤 6 条 → 多样性选取 5 条
```

### 4.4 定时监控 + 变更推送流程

```
用户开启「🔔 分析完成后自动监控」并选择频率

  → SQLite monitor_jobs 表写入记录
  → APScheduler 注册 IntervalTrigger 后台定时任务

       ▼（每周/每天触发）
_async_run_analysis(competitor, our_product)
  1. KnowledgeStore.get_prior_context() → old_context
  2. Orchestrator.run() → new_data（完整新分析）

  → 有变化：写入 change_logs + 触发邮件通知
  → 无变化：跳过写入
       │
  双通道通知：
  ① 页面侧边栏「📡 竞品雷达  🔴 N 条新变更」徽章
  ② SMTP 邮件推送（配置 NOTIFY_* 后自动发送）
```

## 五、各模块详细说明

### 5.1 agents/researcher.py — 研究员 Agent

**核心职责**：并发采集四类数据，构建编号来源，生成带引用的调研摘要

**关键实现**：

```python
# 并发采集（asyncio.gather）
(website_info, website_results), (news_info, news_results), \
(reviews_info, reviews_results), (hiring_info, hiring_results) = \
    await asyncio.gather(
        self._gather_website_info(competitor),
        self._gather_news(competitor),
        self._gather_reviews(competitor),
        self._gather_hiring(competitor),
    )

# 构建编号来源（去重 + 顺序编号）
sources, numbered_header = self._build_numbered_sources(
    website_results, news_results, reviews_results, hiring_results
)

# 后处理：去除 ```markdown 包裹（双重保障）
summary = _strip_code_fence(response.choices[0].message.content)
```

**输出字段**：`website_info / news_info / reviews_info / hiring_info / summary / sources`

### 5.2 agents/evaluator.py — 评估员 Agent（反思循环核心）

6 维度评分（每维 0~2 分，总分 0~12 映射到 1~10）：

| 维度 | 评分标准 |
|------|---------|
| 公司基本信息 | 有具体数字（成立时间/融资/员工）→ 2分 |
| 产品功能描述 | 有差异化优势说明 → 2分 |
| 定价信息 | 有具体价格（如 "$10/月"）→ 2分 |
| 用户评价 | 引用真实用户反馈 → 2分 |
| 市场动态 | 有近期具体事件（时间+内容）→ 2分 |
| SWOT 与战略 | 建议包含具体行动步骤 → 2分 |

- **score ≥ 7**：通过，直接输出报告
- **score < 7**：返回 `gaps`（缺陷说明）+ `supplement_queries`（英文补充查询词），触发 LangGraph `supplement` 节点
- **解析失败**：默认 score=7（通过），避免阻断主流程

### 5.3 core/graph.py — LangGraph 图定义

```
AnalysisState（TypedDict）
  competitor / our_product / prior_context
  research / analysis / product / strategy / evaluation（逐步填入）
  total_tokens / start_time

节点：research → analyst → product → strategist → evaluator
条件边：route_after_eval()
  → "supplement"（score<7 + 有补充查询词）
  → END（score≥7）
supplement → END

编译：graph.compile(checkpointer=MemorySaver())
  每次运行独立 thread_id，支持断点续跑
```

### 5.4 core/report.py — IRDocument 中间表示

```
data dict（各 Agent 原始输出）
    ↓ IRDocument.from_agent_outputs()
IRDocument（结构化章节列表）
    ↓ validate()
  检测各章节是否满足最低字符数（min_chars）
  不足时报告底部追加警告
    ↓ to_markdown()
最终 Markdown 字符串
  含：质量徽章 | 各章节 | 参考来源（可点击）| 薄章节警告
```

**报告质量徽章**：
- 通过：`质量评分：8/10 ✓`
- 已补充：`质量评分：6/10 → 已补充（定价缺失、市场动态模糊）`

**参考来源章节**（报告底部）：
```markdown
## 参考来源
- [1] [Notion Official Website](https://www.notion.so)
- [2] [Notion raises $275M Series C](https://techcrunch.com/...)
```

### 5.5 core/matrix_report.py — 批量对比矩阵报告生成器

**维度提取逻辑**：

| 维度 | 提取方法 |
|------|---------|
| 定价 | regex 匹配 `$N` / `¥N` / `免费` / `Pay-as-you-go` |
| 融资情况 | 逐行匹配「融资 / Series / 估值 / IPO」关键词 |
| 核心优势 | SWOT 优势区域第 1~2 条 bullet |
| 数据质量 | evaluator score，如 `8/10 ✓` |
| 公司概况 | research summary 首段（截取80字） |

**报告结构**：
1. 横向对比矩阵表（N 竞品 × 5 维度）
2. 功能对比矩阵（各竞品 feature_matrix 展开）
3. 各竞品详细摘要（公司概况 + 业务分析 + SWOT）
4. 综合战略建议（如填写我方产品）

---

### 5.6 core/change_detector.py — 变更检测 + 邮件通知

**变更检测**：LLM 对比新旧分析上下文（各取前 3000 字符），只保存有实质变化的记录：

```python
user_msg = f"""
=== 上次分析结论 ===
{old_context[:3000]}

=== 本次分析结论 ===
{new_context[:3000]}"""
```

`_extract_new_context()` 内部字段限制（均已提升以减少截断丢失）：
- summary: 900字 | analysis: 700字 | feature_matrix: 700字 | strategy: 600字

**邮件通知**（`_send_email_notification`）：
- 触发时机：`_save()` 写入 change_logs 后立即调用
- 依赖：标准库 `smtplib`，零额外安装
- 前置条件：`NOTIFY_EMAIL_TO` + `NOTIFY_SMTP_USER` + `NOTIFY_SMTP_PASS` 均已配置
- 容错：发送失败仅打印日志，不影响主流程

---

### 5.7 core/scheduler.py — 定时监控调度器

- **APScheduler BackgroundScheduler**：与 Streamlit 共享进程，@st.cache_resource 单例
- **误触发容错**：`misfire_grace_time=3600`，错过触发时最多 1 小时内补跑
- **重启恢复**：启动时自动从 `monitor_jobs` 表恢复所有启用中的任务
- **线程隔离**：`threading.Lock()` 防止并发分析同一竞品

---

### 5.8 core/knowledge_store.py — 竞品知识库

SQLite `competitor_facts` 表，存储三类事实（每次分析后覆盖更新）：

```sql
CREATE TABLE competitor_facts (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    competitor  TEXT NOT NULL,
    fact_type   TEXT NOT NULL,   -- 'company_overview' / 'business_analysis' / 'feature_matrix'
    content     TEXT NOT NULL,
    updated_at  TEXT
)
```

**增量分析工作流**：
1. 用户输入竞品 → `has_prior_data()` → UI 显示历史数据提示
2. `get_prior_context()` → 格式化为「上次分析结论」文本 → 传入 Orchestrator
3. ResearcherAgent 将 prior_context 注入 LLM prompt → 引导关注变化而非重复历史
4. 分析完成 → `extract_and_save()` → 覆盖保存最新三类事实

## 十一、已知限制与后续改进方向

### 当前已知限制

| 问题 | 说明 | 影响程度 |
|------|------|---------|
| LLM 准确性无法验证 | EvaluatorAgent 只检查结构完整性，不验证数据真实性 | 🟡 中 |
| analyst/strategist 无来源引用 | 只有 researcher summary 有 [N] 引用，分析师结论溯源弱 | 🟡 中 |
| Streamlit 邮件配置需重启生效 | 运行时修改 config.* 后，Scheduler 中已实例化的 detector 不感知新配置 | 🟢 低 |
| 免费模型速率限制 | Gemini Flash 2.0 免费版有并发和每分钟限制，批量分析易触发 429 | 🟡 中 |
| Crawl4AI 安装复杂 | 首次安装需额外步骤，但降级机制完善不影响主流程 | 🟢 低 |

### 后续改进方向

| 方向 | 描述 | 优先级 |
|------|------|-------|
| analyst/strategist 来源引用 | 将 researcher sources 传递给下游 Agent，引导 LLM 在分析中引用 [N] | P3 |
| Webhook 通知 | 支持 Slack/钉钉/飞书 Webhook，覆盖无邮件的团队场景 | P3 |
| 首次使用引导 | 检测 API Key 缺失时弹出分步配置 Onboarding，降低上手门槛 | P3 |
| 报告模板自定义 | 允许用户调整章节顺序、添加自定义维度 | P4 |
| 多语言报告 | 支持生成英文版报告，面向国际化团队 | P4 |
